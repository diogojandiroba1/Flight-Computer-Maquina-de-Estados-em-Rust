Manual Teórico — Flight Computer em Rust

Este manual cobre cada conceito de Rust que você precisa dominar para construir o projeto, na ordem em que eles constroem uns sobre os outros. Cada seção explica o porquê antes do como.

Por que Rust para sistemas críticos?

Rust foi desenhado com uma garantia central: se o código compila, ele não tem race conditions nem use-after-free. Isso é verificado pelo compilador em tempo de compilação, sem runtime overhead. Para sistemas aeroespaciais — onde um bug pode significar perda de missão — isso não é conveniência, é requisito.

Zero-cost abstractions Você escreve código de alto nível. O compilador gera o equivalente ao C otimizado. Sem garbage collector, sem runtime.
Ownership system A memória é gerenciada pelo compilador via regras de propriedade. Sem malloc/free, sem dangling pointers.
Type system expressivo Estados inválidos do sistema podem ser literalmente impossíveis de expressar no código. O typestate pattern vai fazer isso.
Fearless concurrency O compilador garante que não há data races entre threads. Race conditions viram erros de compilação.

Ownership, Borrowing e Move Semantics

Ownership é a fundação de tudo em Rust. Sem entender ownership, nenhum outro conceito faz sentido completamente. É o sistema que elimina toda uma classe de bugs sem um garbage collector.

A regra fundamental

Em Rust, cada valor tem exatamente um dono (owner). Quando o owner sai de escopo, o valor é destruído. Isso é garantido pelo compilador, não pelo programador.

fn main() {
    let altitude = 12000.0_f64; // altitude é o dono desse f64
    
    // altitude existe aqui
    println!("{}", altitude);
    
} // altitude sai de escopo → memória liberada automaticamente

Move semantics — transferência de propriedade

Quando você atribui um valor a outra variável, a propriedade é movida (moved). A variável original fica inválida — o compilador rejeita qualquer uso posterior dela.

let dados = String::from("dados de telemetria");
// dados é dono da String

let novos_dados = dados; // propriedade MOVIDA para novos_dados

println!("{}", dados); // ❌ ERRO DE COMPILAÇÃO: value moved here
                          //    dados não existe mais
Para o Flight Computer, isso é crucial: quando você chama rocket.ignite(), o estado PreLaunch é consumido e nunca mais pode ser usado. O compilador garante isso.

Borrowing — empréstimo sem transferência

Você pode emprestar um valor sem transferir a propriedade, usando referências (&). Existem dois tipos de empréstimo:

let telemetria = String::from("altitude: 12000m");

// Empréstimo imutável — múltiplos permitidos ao mesmo tempo
let ref1 = &telemetria;
let ref2 = &telemetria;
println!("{} | {}", ref1, ref2); // ✅ OK

// Empréstimo mutável — apenas UM de cada vez
let mut dados = String::from("dados");
let r_mut = &mut dados;
// não pode usar dados aqui enquanto r_mut existe

A regra do borrow checker

Referências imutáveis (&T) Você pode ter quantas quiser ao mesmo tempo. Nenhuma pode modificar o valor. Útil para leitura.
Referência mutável (&mut T) Apenas uma por vez. Nenhuma outra referência pode existir enquanto ela está ativa. Garante exclusividade de escrita.

Copy vs Move

Tipos primitivos como f64, i32, bool implementam o trait Copy — eles são copiados automaticamente ao invés de movidos. Tipos que alocam memória (como String e Vec) são movidos.

let altitude: f64 = 12000.0;   // f64 implementa Copy
let copia = altitude;              // CÓPIA — altitude ainda válida
println!("{}", altitude);          // ✅ funciona

let nome = String::from("MECO"); // String não implementa Copy
let movido = nome;                 // MOVE — nome inválido
// println!("{}", nome);           // ❌ erro

Lifetimes — o que o borrow checker rastreia

Lifetimes são o mecanismo interno que Rust usa para rastrear por quanto tempo uma referência é válida. Na maioria dos casos eles são inferidos. Você os verá explicitamente em structs que guardam referências.

// Lifetime explícita: a struct não pode outlive a referência que ela guarda
struct TelemetrySlice<'a> {
    dados: &'a str, // 'a diz: essa referência deve viver pelo menos tanto quanto a struct
}

// No flight computer, prefira possuir os dados (String em vez de &str)
// para evitar lifetime complexity no início.
Para o Flight Computer: os estados do foguete serão structs simples sem referências, então lifetimes explícitas não serão necessárias no início. O move semantics do typestate pattern é a parte importante aqui.

Sistema de Tipos — Structs e Generics

O sistema de tipos de Rust é a base do typestate pattern. Entender structs e generics é obrigatório antes de modelar os estados do foguete.

Structs — dados agrupados

Uma struct agrupa campos relacionados. Em Rust, structs não têm herança — comportamento vem de traits implementadas sobre elas.

// Struct com campos nomeados
struct TelemetryData {
    altitude:  f64,    // altitude em metros
    velocity:  f64,    // velocidade em m/s
    timestamp: u64,    // milissegundos desde ignição
}

// Instanciação — todos os campos obrigatórios
let dados = TelemetryData {
    altitude:  45000.0,
    velocity:  800.0,
    timestamp: 120_000,
};

// Acesso por campo
println!("Alt: {}m", dados.altitude);

Struct sem campos — marcador de estado

Uma zero-size struct existe apenas como tipo — ocupa zero bytes em memória. Isso é exatamente o que usaremos para marcar estados do foguete.

// Tipos marcadores — só existem como tipos, sem dados
struct PreLaunch;   // zero bytes
struct Ignition;    // zero bytes
struct MaxQ;         // zero bytes
struct MECO;         // zero bytes
struct Separation;  // zero bytes
struct Orbit;        // zero bytes

// No binário final, esses tipos não existem como dados.
// Eles existem apenas no sistema de tipos do compilador.

Generics — parâmetros de tipo

Generics permitem que uma struct ou função seja parametrizada por um tipo. O compilador gera código especializado para cada tipo concreto usado — isso é monomorphization, sem overhead de runtime.

// <T> é o parâmetro de tipo — pode ser qualquer tipo
struct Rocket<T> {
    state:      T,        // campo com tipo genérico
    mass_kg:    f64,
    fuel_level: f64,
}

// Rocket<PreLaunch> e Rocket<Orbit> são tipos COMPLETAMENTE DIFERENTES
// para o compilador — tão diferentes quanto i32 e String

let foguete: Rocket<PreLaunch> = Rocket {
    state:      PreLaunch,
    mass_kg:    549_000.0,
    fuel_level: 1.0,
};

impl blocks com generics

Você pode implementar métodos apenas para versões específicas de um tipo genérico. Isso é o mecanismo central do typestate pattern.

// Métodos para QUALQUER estado — acessíveis sempre
impl<T> Rocket<T> {
    pub fn fuel_level(&self) -> f64 {
        self.fuel_level
    }
}

// Métodos APENAS para Rocket<PreLaunch>
impl Rocket<PreLaunch> {
    pub fn new() -> Self {
        Rocket { state: PreLaunch, mass_kg: 549_000.0, fuel_level: 1.0 }
    }
    // ignite() só existe neste tipo — não em Rocket<Orbit>
    pub fn ignite(self) -> Rocket<Ignition> { /* ... */ }
}

// Métodos APENAS para Rocket<MECO>
impl Rocket<MECO> {
    pub fn separate_stage(self) -> Rocket<Separation> { /* ... */ }
}
Note que ignite(self) recebe self por valor, não por referência. Isso consome o Rocket<PreLaunch>. Depois de chamar ignite(), o foguete pré-lançamento literalmente não existe mais — apenas o foguete em ignição existe.

PhantomData — quando o compilador precisa de ajuda

Se o parâmetro genérico não aparece em nenhum campo real da struct, o compilador reclama que o tipo não é usado. PhantomData resolve isso: informa ao compilador que o tipo está "conceitualmente presente" sem ocupar espaço.

use std::marker::PhantomData;

// Se você quiser separar o estado dos dados do foguete:
struct Rocket<State> {
    mass_kg:    f64,
    fuel_level: f64,
    _state:     PhantomData<State>, // zero bytes, satisfaz o compilador
}

// PhantomData<State> não ocupa memória.
// É uma instrução para o compilador: "este tipo existe aqui conceitualmente".

Typestate Pattern

O typestate pattern é a técnica central do projeto. Ele transforma estados de runtime em estados de compilação, tornando transições inválidas literalmente impossíveis de escrever.

O problema que ele resolve

Em linguagens convencionais, validação de estado acontece em runtime:

Abordagem comum (runtime check)
// Python/Java — estado como enum em runtime
class Rocket:
    state = "PreLaunch"
    
    def separate_stage(self):
        # Verificado em RUNTIME
        if self.state != "MECO":
            raise Exception("Estado inválido!")
        # O bug pode existir no código
        # e só é descoberto ao executar
Typestate em Rust (compile-time)
// Rust — estado no sistema de tipos
let rocket = Rocket::<PreLaunch>::new();

// separate_stage() não existe em
// Rocket<PreLaunch> — não tem como chamar
rocket.separate_stage();
// ❌ ERRO DE COMPILAÇÃO:
// no method `separate_stage` found
// for type `Rocket<PreLaunch>`
// O bug é impossível de escrever.

Implementação completa

A seguir a implementação completa dos estados e transições, linha a linha:

// ── src/states/mod.rs ──────────────────────────────────────────────

// Tipos marcadores — zero bytes, apenas existem como tipos
pub struct PreLaunch;
pub struct Ignition;
pub struct MaxQ;
pub struct MECO;
pub struct Separation;
pub struct Orbit;
pub struct Aborted;  // estado final de falha

// ── src/rocket.rs ──────────────────────────────────────────────────

use std::marker::PhantomData;
use crate::states::*;

// A struct central — parametrizada pelo estado
pub struct Rocket<State> {
    pub mass_kg:      f64,
    pub fuel_level:   f64,   // 0.0 a 1.0
    pub altitude_m:   f64,
    pub velocity_ms:  f64,
    _state: PhantomData<State>,  // zero bytes
}

// Métodos disponíveis em QUALQUER estado
impl<State> Rocket<State> {
    // Transição interna — usada pelas funções de transição
    fn transition<NewState>(self) -> Rocket<NewState> {
        Rocket {
            mass_kg:     self.mass_kg,
            fuel_level:  self.fuel_level,
            altitude_m:  self.altitude_m,
            velocity_ms: self.velocity_ms,
            _state:      PhantomData,
        }
    }
}

// ── Estado: PreLaunch ──────────────────────────────────────────────
impl Rocket<PreLaunch> {
    pub fn new() -> Self {
        Rocket {
            mass_kg:     549_054.0,  // Falcon 9 LOX+RP-1
            fuel_level:  1.0,
            altitude_m:  0.0,
            velocity_ms: 0.0,
            _state:      PhantomData,
        }
    }

    // ignite(self) — consome o PreLaunch, produz Ignition
    // self por VALOR = move semântico = PreLaunch é destruído
    pub fn ignite(self) -> Rocket<Ignition> {
        log::info!("Ignição iniciada — T+0");
        self.transition()
    }
}

// ── Estado: MECO ───────────────────────────────────────────────────
impl Rocket<MECO> {
    // separate_stage SÓ EXISTE aqui — não existe em outros estados
    pub fn separate_stage(self) -> Rocket<Separation> {
        log::info!("Separação de estágio");
        self.transition()
    }
}

Por que self e não &self?

Este é o detalhe mais importante do typestate pattern. Quando um método recebe self (valor), ele consome a instância. Depois da chamada, o foguete no estado antigo literalmente não existe mais na memória do programa. O compilador rejeita qualquer tentativa de usá-lo.

let pre_launch = Rocket::<PreLaunch>::new();

// ignite() consome pre_launch e retorna um NOVO objeto
let igniting = pre_launch.ignite();  // pre_launch foi movido

// Qualquer uso de pre_launch agora é erro de compilação:
// println!("{}", pre_launch.fuel_level);
// ❌ error[E0382]: borrow of moved value: `pre_launch`

// O sistema de tipos garante a progressão linear:
// PreLaunch → Ignition → MaxQ → MECO → Separation → Orbit

Abort — transição de qualquer estado

O abort precisa ser possível a partir de vários estados. O jeito idiomático em Rust é usar um trait:

// Trait que qualquer estado abortável deve implementar
pub trait Abortable {
    fn abort(self) -> Rocket<Aborted>;
}

// Implementar para os estados que podem abortar
impl Abortable for Rocket<Ignition> {
    fn abort(self) -> Rocket<Aborted> {
        log::error!("ABORT — fase Ignition");
        self.transition()
    }
}

Enums com Dados e Pattern Matching

Os enums de Rust são tipos soma — cada variante pode carregar dados completamente diferentes. Eles modelam o mundo real melhor que enums de outras linguagens.

Enums simples vs enums com dados

Enum simples (sem dados)
// Apenas discrimina entre casos
enum FlightPhase {
    PreLaunch,
    Ignition,
    Orbit,
}
// Não carrega dados específicos
// da fase — não muito útil
Enum com dados (rico)
// Cada variante carrega seus dados
enum FlightEvent {
    SensorReading {
        altitude:  f64,
        velocity:  f64,
        timestamp: u64,
    },
    FaultDetected {
        subsystem: Subsystem,
        severity:  Severity,
    },
    StageCompleted {
        stage:      FlightStage,
        elapsed_ms: u64,
    },
}

O enum completo do projeto

// ── src/events.rs ──────────────────────────────────────────────────

use crate::subsystems::{Subsystem, Severity};
use crate::states::FlightStage;

pub enum FlightEvent {
    // Leitura periódica de sensores
    SensorReading {
        altitude:  f64,
        velocity:  f64,
        timestamp: u64,
    },

    // Falha detectada em subsistema
    FaultDetected {
        subsystem: Subsystem,
        severity:  Severity,
        message:   String,
    },

    // Fase completada com sucesso
    StageCompleted {
        stage:      FlightStage,
        elapsed_ms: u64,
    },

    // Comando de abort recebido
    AbortCommand {
        reason: String,
        source: Subsystem,
    },

    // Telemetria de propulsão
    ThrustReading(f64),  // variante com dado tupla
}

// Subsistemas que podem reportar eventos
pub enum Subsystem {
    Propulsion,
    Navigation,
    Telemetry,
    PowerSystem,
}

pub enum Severity {
    Warning,   // monitorar
    Critical,  // considerar abort
    Fatal,     // abort imediato
}

Pattern matching com match

O match é a ferramenta para trabalhar com enums. Rust exige que todos os casos sejam cobertos — o compilador rejeita código com casos não tratados.

// Processar um evento recebido pelo flight computer
fn processar_evento(evento: FlightEvent) {
    match evento {
        // Destruturar campos nomeados
        FlightEvent::SensorReading { altitude, velocity, timestamp } => {
            log::info!("T+{}: alt={:.0}m vel={:.1}m/s", timestamp, altitude, velocity);
            
            // Checar condição de MaxQ (pressão dinâmica máxima ~85km)
            if altitude > 80_000.0 && altitude < 90_000.0 {
                log::warn!("Aproximando MaxQ — pressão dinâmica alta");
            }
        }

        // Falha — usar guard para checar severidade
        FlightEvent::FaultDetected { subsystem, severity: Severity::Fatal, message } => {
            log::error!("FALHA FATAL em {:?}: {}", subsystem, message);
            // acionar abort
        }

        FlightEvent::FaultDetected { subsystem, severity, message } => {
            log::warn!("Falha em {:?} ({:?}): {}", subsystem, severity, message);
        }

        FlightEvent::StageCompleted { stage, elapsed_ms } => {
            log::info!("Fase {:?} concluída em {}ms", stage, elapsed_ms);
        }

        // _ cobre todos os casos não listados explicitamente
        _ => {}
    }
}

if let — match para um caso só

Quando você só se importa com uma variante específica, if let é mais conciso que um match completo.

let evento = FlightEvent::ThrustReading(7_607_000.0);

// match para um caso só
if let FlightEvent::ThrustReading(thrust) = evento {
    println!("Empuxo: {:.0}N", thrust);
}

// while let — processar canal até estar vazio
while let Ok(evento) = rx.try_recv() {
    processar_evento(evento);
}

Option<T> — ausência segura de valores

Option<T> é um enum da biblioteca padrão que representa "pode ter um valor, pode não ter". É o substituto de null em Rust — o compilador força você a tratar ambos os casos.

enum Option<T> {  // definição real na stdlib
    Some(T),  // há um valor do tipo T
    None,    // sem valor
}

// Sensores podem retornar leitura inválida
fn ler_gps() -> Option<f64> {
    if sinal_disponivel() {
        Some(12000.0)
    } else {
        None  // sem sinal — sem null pointer
    }
}

// Tratar o Option com match
match ler_gps() {
    Some(altitude) => println!("GPS: {}m", altitude),
    None           => println!("Sinal GPS perdido — usando INS"),
}

Traits — Comportamento Abstrato

Traits definem um conjunto de métodos que um tipo deve implementar. São o mecanismo de polimorfismo em Rust — mais poderoso que herança porque são composáveis.

Definindo e implementando um trait

// ── src/sensors.rs ────────────────────────────────────────────────

// Trait: contrato que qualquer sensor deve cumprir
pub trait Sensor {
    // Tipo associado — cada sensor define seu próprio tipo de leitura
    type Reading;

    // Método obrigatório — todo sensor deve implementar
    fn read(&self) -> Result<Self::Reading, SensorError>;

    // Método com implementação padrão — pode ser sobrescrito
    fn name(&self) -> &str {
        "sensor desconhecido"
    }

    // Método padrão que usa read() — composição
    fn is_healthy(&self) -> bool {
        self.read().is_ok()
    }
}

// Altímetro — implementa o trait Sensor
pub struct Altimeter {
    calibration_offset: f64,
}

impl Sensor for Altimeter {
    type Reading = f64;  // altitude em metros

    fn read(&self) -> Result<f64, SensorError> {
        // Simular leitura com ruído
        let base = 45000.0_f64;
        Ok(base + self.calibration_offset)
    }

    fn name(&self) -> &str { "Altímetro barométrico" }
}

// GPS — outro tipo, mesma interface
pub struct GPS {
    signal_strength: f64,
}

impl Sensor for GPS {
    type Reading = (f64, f64, f64);  // (lat, lon, altitude)

    fn read(&self) -> Result<(f64, f64, f64), SensorError> {
        if self.signal_strength < 0.3 {
            Err(SensorError::NoSignal)
        } else {
            Ok((-23.0, -46.0, 45000.0))
        }
    }
}

Trait bounds — generics com restrições

Você pode exigir que um tipo genérico implemente um trait específico. O compilador verifica isso e garante que todos os métodos do trait estão disponíveis.

// T deve implementar Sensor — garantido pelo compilador
fn coletar_dados<T: Sensor>(sensor: &T) -> String {
    match sensor.read() {
        Ok(_)  => format!("{}: OK", sensor.name()),
        Err(e) => format!("{}: FALHA — {:?}", sensor.name(), e),
    }
}

// Múltiplos bounds com + (AND lógico)
fn sensor_com_log<T: Sensor + std::fmt::Debug>(sensor: &T) {
    println!("{:?}", sensor);
}

// Sintaxe alternativa com where (mais legível para muitos bounds)
fn init_sensor<T>(sensor: &T)
where
    T: Sensor + Send + Sync,  // Send+Sync para uso em threads
{ /* ... */ }

Trait objects — polimorfismo dinâmico

Quando você quer uma coleção de sensores de tipos diferentes (altímetro, GPS, acelerômetro), use dyn Trait. Isso usa dynamic dispatch (vtable), diferente dos generics que são monomorphized.

// Vec heterogêneo de sensores — tipos diferentes, mesma interface
let sensores: Vec<Box<dyn Sensor<Reading = f64>>> = vec![
    Box::new(Altimeter { calibration_offset: 0.0 }),
    Box::new(Barometer { base_pressure: 101325.0 }),
];

// Iterar — cada sensor pode ser um tipo diferente
for sensor in &sensores {
    match sensor.read() {
        Ok(v)  => println!("{}: {:.1}m", sensor.name(), v),
        Err(e) => println!("{}: erro {:?}", sensor.name(), e),
    }
}

// Box<dyn Trait> = ponteiro heap para objeto de tamanho desconhecido em compile time
// O compilador não sabe o tamanho exato → precisa ser boxed

Result<T, E> e Tratamento de Erros

Rust não tem exceptions. Erros são valores — tratados com Result<T, E>. O compilador força o tratamento explícito de cada falha possível. Em sistemas críticos, isso é uma propriedade, não um inconveniente.

O tipo Result

// Definição real na stdlib:
enum Result<T, E> {
    Ok(T),   // sucesso — carrega o valor do resultado
    Err(E),  // falha — carrega o erro
}

// No flight computer — toda operação crítica retorna Result
fn verificar_preflight() -> Result<(), FlightError> {
    if nivel_combustivel() < 0.95 {
        return Err(FlightError::InsufficientFuel);
    }
    if !sistemas_online() {
        return Err(FlightError::SystemOffline { 
            system: Subsystem::Navigation 
        });
    }
    Ok(())  // () = "unit" — sucesso sem valor
}

Definindo erros com thiserror

A crate thiserror elimina boilerplate ao derivar implementações de std::error::Error automaticamente.

// ── src/errors.rs ─────────────────────────────────────────────────

use thiserror::Error;

// #[derive(Error)] implementa std::error::Error automaticamente
// #[error("...")] define a mensagem de display
#[derive(Debug, Error)]
pub enum FlightError {
    #[error("combustível insuficiente para lançamento")]
    InsufficientFuel,

    #[error("sistema {system:?} offline antes do lançamento")]
    SystemOffline { system: Subsystem },

    #[error("falha de sensor: {0}")]  // {0} = primeiro campo tuple
    SensorFailure(String),

    #[error("transição de estado inválida: {from} → {to}")]
    InvalidTransition { from: String, to: String },

    // Wrapping de erros de outras crates com #[from]
    #[error("erro de I/O: {0}")]
    Io(#[from] std::io::Error),
}

// Erros de sensor — separados por domínio
#[derive(Debug, Error)]
pub enum SensorError {
    #[error("sem sinal")]          NoSignal,
    #[error("leitura fora do range")] OutOfRange,
    #[error("sensor não inicializado")] NotInitialized,
}

O operador ? — propagação de erros

O operador ? é açúcar sintático: se o Result é Err, retorna o erro imediatamente da função atual. Elimina o boilerplate de checar cada resultado manualmente.

// Sem o operador ? — verboso
fn preflight_manual() -> Result<(), FlightError> {
    let fuel = match verificar_combustivel() {
        Ok(f)  => f,
        Err(e) => return Err(e),
    };
    let nav = match verificar_navegacao() {
        Ok(n)  => n,
        Err(e) => return Err(e),
    };
    Ok(())
}

// Com o operador ? — idiomático
fn preflight() -> Result<(), FlightError> {
    let fuel = verificar_combustivel()?;  // retorna Err se falhar
    let nav  = verificar_navegacao()?;    // retorna Err se falhar
    Ok(())  // chegou aqui = tudo OK
}

// ? também converte erros automaticamente via From trait
// std::io::Error é convertido para FlightError::Io graças ao #[from]
fn salvar_telemetria(dados: &str) -> Result<(), FlightError> {
    std::fs::write("telemetry.log", dados)?;  // io::Error → FlightError::Io
    Ok(())
}

Threads e Channels — Concorrência

Subsistemas reais de voo rodam em paralelo. Rust permite modelar isso com threads que se comunicam por channels, com garantias de segurança verificadas em tempo de compilação.

thread::spawn — criar uma thread

use std::thread;
use std::time::Duration;

// spawn aceita uma closure que será executada na nova thread
let handle = thread::spawn(|| {
    // Este código roda em paralelo com a thread principal
    for i in 0..5 {
        println!("Propulsão: ciclo {}", i);
        thread::sleep(Duration::from_millis(100));
    }
});

// join() bloqueia até a thread terminar e retorna Result
// (pode falhar se a thread entrou em pânico)
handle.join().expect("thread de propulsão falhou");

mpsc::channel — comunicação entre threads

mpsc = Multiple Producer, Single Consumer. Vários subsistemas (producers) enviam eventos para o flight computer central (consumer).

use std::sync::mpsc;
use std::thread;

// Criar canal — tx = transmitter, rx = receiver
let (tx, rx) = mpsc::channel::<FlightEvent>();

// MÚLTIPLOS producers — clonar o tx para cada subsistema
let tx_propulsao   = tx.clone();
let tx_navegacao   = tx.clone();
let tx_telemetria  = tx.clone();

// Thread de propulsão — move tx_propulsao para dentro
thread::spawn(move || {
    // move keyword: transfere tx_propulsao para a closure
    loop {
        let empuxo = ler_sensor_empuxo();
        tx_propulsao.send(FlightEvent::ThrustReading(empuxo))
            .expect("canal fechado");
        thread::sleep(Duration::from_millis(10));
    }
});

// Thread de navegação
thread::spawn(move || {
    loop {
        let (alt, vel) = ler_inercial();
        tx_navegacao.send(FlightEvent::SensorReading {
            altitude:  alt,
            velocity:  vel,
            timestamp: now_ms(),
        }).expect("canal fechado");
        thread::sleep(Duration::from_millis(50));
    }
});

// ── Thread principal — flight computer ────────────────────────────
// Loop de processamento de eventos
loop {
    // recv() bloqueia até chegar um evento
    match rx.recv() {
        Ok(evento)  => processar_evento(evento),
        Err(_)      => {
            // Err significa que todos os tx foram dropped
            // = todos os subsistemas terminaram
            println!("Todos os subsistemas encerrados");
            break;
        }
    }
}

O que o compilador garante

Rust garante em compile time que você não pode enviar um tipo não-Send por um canal para outra thread. Se tentar enviar um Rc<T> (que não é thread-safe) para outra thread, o compilador rejeita com erro. Isso elimina toda uma classe de bugs de concorrência antes de rodar o código. 
// Rc não implementa Send — o compilador rejeita isso:
use std::rc::Rc;
let (tx, _rx) = mpsc::channel();
let rc = Rc::new(42);

thread::spawn(move || {
    tx.send(rc).unwrap(); // ❌ ERRO: Rc cannot be sent between threads safely
                           // Use Arc ao invés de Rc
});

Arc<Mutex<T>> — Estado Compartilhado

Quando múltiplas threads precisam ler e escrever no mesmo dado, usamos Arc<Mutex<T>>. Arc gerencia a vida útil do dado. Mutex garante acesso exclusivo. Juntos, eliminam race conditions.

Rc vs Arc — contagem de referência

Rc<T> Reference Counting — para uma única thread. Não é Send. Overhead zero de sincronização.
Arc<T> Atomic Reference Counting — thread-safe. Implementa Send+Sync. Pequeno overhead de operações atômicas. 
use std::sync::{Arc, Mutex};
use std::thread;

// ── Como Arc funciona ──────────────────────────────────────────────
// Arc = ponteiro compartilhado com contagem de referência atômica
// Todos os Arc para o mesmo dado compartilham o mesmo valor na heap
// Quando o último Arc é dropped, o valor é liberado

let dados = Arc::new(42);
let clone1 = Arc::clone(&dados); // incrementa contagem (=2)
let clone2 = Arc::clone(&dados); // incrementa contagem (=3)

// clone1 e clone2 podem ir para threads diferentes
// todos apontam para o mesmo 42 na heap
drop(clone1); // contagem = 2
drop(clone2); // contagem = 1
// dados ainda válido — contagem = 1
// quando dados sai de escopo: contagem = 0 → liberado

Mutex — acesso exclusivo

// Mutex garante que apenas uma thread acessa o dado por vez
// lock() bloqueia até o Mutex estar disponível

let mutex = Mutex::new(0_i32); // Mutex encapsula o dado

{
    // lock() retorna MutexGuard — acesso ao dado interno
    // O compilador garante que você usa o dado ATRAVÉS do guard
    let mut guard = mutex.lock().unwrap();
    *guard += 1;
    // guard é dropped aqui → Mutex liberado automaticamente
    // Não há como "esquecer" de liberar o lock
}

// Se outra thread tiver o lock, lock() bloqueia até ela terminar

Arc<Mutex<T>> no Flight Computer

// Estado global compartilhado entre subsistemas
#[derive(Debug)]
pub struct FlightState {
    pub altitude_m:   f64,
    pub velocity_ms:  f64,
    pub fuel_level:   f64,
    pub abort_flag:   bool,
}

// Criar estado compartilhado
let state = Arc::new(Mutex::new(FlightState {
    altitude_m:  0.0,
    velocity_ms: 0.0,
    fuel_level:  1.0,
    abort_flag:  false,
}));

// Dar uma cópia do Arc para cada thread
let state_nav  = Arc::clone(&state);
let state_prop = Arc::clone(&state);

// Thread de navegação — atualiza altitude e velocidade
thread::spawn(move || {
    loop {
        let (nova_alt, nova_vel) = calcular_inercial();
        
        // Adquirir lock, modificar, liberar automaticamente
        let mut s = state_nav.lock().unwrap();
        s.altitude_m  = nova_alt;
        s.velocity_ms = nova_vel;
        // lock liberado aqui — } fim do escopo do guard
    }
});

// Thread de propulsão — consome combustível
thread::spawn(move || {
    loop {
        let mut s = state_prop.lock().unwrap();
        s.fuel_level -= 0.001;
        
        // Detectar abort — setar flag para outras threads
        if s.fuel_level < 0.05 {
            s.abort_flag = true;
            break;
        }
    }
});

// Thread principal — monitorar abort flag
loop {
    let s = state.lock().unwrap();
    if s.abort_flag {
        break; // acionar sequência de abort
    }
    // lock liberado aqui — não seguramos durante sleep
    drop(s);
    thread::sleep(Duration::from_millis(100));
}
Nunca mantenha um lock enquanto chama código que pode bloquear (como recv() ou operações de I/O). Isso causa deadlock. O padrão é: adquirir lock → modificar → dropar imediatamente → continuar.

use std::sync::{Arc, Mutex};
use std::sync::mpsc;
use crate::events::FlightEvent;
use crate::states::PreLaunch;

fn main() -> Result<(), crate::errors::FlightError> {
    env_logger::init();
    log::info!("=== FLIGHT COMPUTER v0.1 ===");

    // Estado compartilhado
    let shared_state = Arc::new(Mutex::new(
        FlightState::default()
    ));

    // Canal de eventos
    let (tx, rx) = mpsc::channel::<FlightEvent>();

    // Iniciar subsistemas
    subsystems::propulsion::start(tx.clone(), Arc::clone(&shared_state));
    subsystems::navigation::start(tx.clone(), Arc::clone(&shared_state));
    subsystems::telemetry::start(tx.clone(), Arc::clone(&shared_state));

    // Criar foguete no estado inicial
    let rocket = Rocket::<PreLaunch>::new();
    
    // Preflight check
    preflight_check(&rocket)?;  // ? propaga FlightError se falhar

    // Sequência de lançamento — cada linha consome o estado anterior
    let rocket = rocket.ignite();    // PreLaunch → Ignition
    let rocket = rocket.max_q();     // Ignition → MaxQ
    let rocket = rocket.meco();      // MaxQ → MECO
    let rocket = rocket.separate(); // MECO → Separation
    let _orbit = rocket.achieve_orbit(); // Separation → Orbit

    log::info!("=== ÓRBITA ALCANÇADA ===");
    Ok(())
}

Padrão de cada subsistema

// ── src/subsystems/propulsion.rs ───────────────────────────────────
use std::sync::{Arc, Mutex};
use std::sync::mpsc::Sender;
use std::thread;
use crate::events::FlightEvent;

// Função pública para iniciar o subsistema em uma nova thread
pub fn start(tx: Sender<FlightEvent>, state: Arc<Mutex<FlightState>>) {
    thread::Builder::new()
        .name("propulsion".into())  // nome da thread para debug
        .spawn(move || run_loop(tx, state))
        .expect("failed to spawn propulsion thread");
}

fn run_loop(tx: Sender<FlightEvent>, state: Arc<Mutex<FlightState>>) {
    let mut rng = rand::thread_rng();
    let cycle = std::time::Duration::from_millis(10);

    loop {
        // Verificar abort flag sem manter o lock
        let should_abort = state.lock().unwrap().abort_flag;
        if should_abort { break; }

        // Simular leitura de empuxo
        let thrust = 7_607_000.0 + rng.gen_range(-5000.0..5000.0);
        
        // Enviar evento — se o receiver foi dropped, encerrar
        if tx.send(FlightEvent::ThrustReading(thrust)).is_err() {
            break;
        }

        thread::sleep(cycle);
    }
    log::info!(target: "propulsion", "Thread encerrada");
}
